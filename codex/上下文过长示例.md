# 上下文过长时的摘要示例

原始对话大概长这样：

- user：帮我把 `A` 接口改成支持分页
- assistant：先看代码结构，发现分页入口在 `service/list.rs`
- user：顺便把返回字段从 `items` 改成 `data`
- assistant：已经改了 3 个文件
- user：再补一个测试，覆盖空列表
- assistant：我去补测试
- user：继续追问一些细节，后面又聊了很多，导致上下文快满了

这时触发压缩后，系统会让模型生成一段 handoff summary，例如：

```text
- Current progress: `A` 接口已改造到分页版，返回字段准备从 `items` 切到 `data`
- Key decisions: 使用游标分页，保持兼容旧调用方
- Important context: 相关逻辑集中在 `service/list.rs`，测试要覆盖空列表和边界页
- Remaining work: 补测试、检查文档、确认前端联调字段名
```

然后系统会把这段摘要放回压缩后的历史里，作为新的“上下文前半段”，同时通常还会保留最近几条真实 user 消息。效果就是：

- 老的长对话被折叠掉一部分
- 关键结论被保留下来
- 模型接下来继续干活时，能接着这份摘要往下做
