# 上下文压缩与历史检索分析

## 结论

1. 当前实现里，没有看到“压缩后自动去检索旧对话，再把补回信息注入当前上下文”的闭环。
2. 压缩后，模型当前看到的是压缩后的历史和摘要，不再是原始完整对话，所以如果只依赖模型记忆去恢复之前的文件，稳定性不高。
3. 真正可靠的恢复路径，是去读本地持久化的 rollout / jsonl 历史，而不是依赖压缩后的上下文。
4. 终端启动和 app-server 启动在这点上本质一致：模型没有一个专门的“rollout 查询工具”，历史检索是外部逻辑完成的。

## 压缩输入是什么

- 本地/内联压缩：输入是当前 session 里已经积累的 history，再加一条压缩提示。
- remote compaction v2：输入是当前 history，必要时先裁剪输出，再附加 `CompactionTrigger`。
- 都不是回扫全量历史后再做检索式补全。

## 压缩输出是什么样

- 本地/内联压缩：会生成一个交接摘要，前面带固定前缀 `SUMMARY_PREFIX`，后面是模型写出的摘要正文。
- remote compaction v2：会生成结构化的 `ResponseItem::Compaction`，再被写回新的 history。

## 历史线程搜索

- `thread/search` 和类似的列表搜索，底层会扫 rollout / jsonl。
- 这类搜索是给客户端/API 用的，不是模型自己直接调用的工具。
- 目前模型可见的 `tool_search` 只用于 deferred tools 元数据搜索，不用于 rollout/history 搜索。

## 对“恢复之前修改的文件”的影响

- 如果用户在压缩后隔了几轮再问“之前改了什么文件”，单靠压缩后的上下文不稳。
- 更稳的办法是：从 rollout/jsonl 里查工具调用、patch 事件、diff、日志，再结合文件系统或 git 状态恢复。
- 如果改动来自 `apply_patch`、patch diff 或受控工具调用，rollout 里通常有足够线索；如果是手工 shell 修改，就要额外依赖文件系统或 git。
