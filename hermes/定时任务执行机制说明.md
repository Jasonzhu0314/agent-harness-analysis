# Hermes 定时任务执行机制说明

## 结论

Hermes 的定时任务不是系统 `crontab` 直接执行，而是 Hermes 自己的 cron 子系统：

- 任务配置保存在 `~/.hermes/cron/jobs.json`
- gateway 启动后创建后台 `cron-scheduler` 线程
- 默认每 60 秒调用一次 `cron.scheduler.tick()`
- 到期任务会被提交到线程池执行
- 执行结果会保存为 cron 独立输出，并可选投递到聊天平台

## 执行链路

```text
hermes cron create / cronjob tool
  -> cron.jobs.create_job()
  -> 写入 ~/.hermes/cron/jobs.json，计算 next_run_at
  -> gateway 启动 cron-scheduler 线程
  -> InProcessCronScheduler.start() 每 60s 调 cron.scheduler.tick()
  -> tick() 加 .tick.lock 防重复执行
  -> get_due_jobs() 找到 next_run_at <= now 的任务
  -> advance_next_run() 先推进 recurring 任务，避免崩溃后重复补跑
  -> 提交到线程池
  -> run_one_job()
  -> run_job()
  -> save_job_output()
  -> _deliver_result()
  -> mark_job_run()
```

关键代码位置：

- `cron/jobs.py`：任务创建、存储、schedule 解析、`next_run_at` 维护
- `cron/scheduler_provider.py`：默认 60 秒 ticker
- `gateway/run.py`：gateway 启动 cron scheduler 线程
- `cron/scheduler.py`：到期判定、执行、保存输出、投递结果、更新状态
- `tools/cronjob_tools.py`：模型侧统一 cron 管理工具
- `hermes_cli/cron.py`：CLI 管理入口

## 与主 Agent 的隔离

定时任务和主 agent 会话是隔离的，但不是操作系统进程级完全隔离。

隔离点：

- 每次 cron 执行都会新建独立 `AIAgent`
- `platform="cron"`
- `session_id` 形如 `cron_{job_id}_{timestamp}`
- 不传主会话的 `conversation_history`
- 会清空 gateway 的 `HERMES_SESSION_*` 上下文变量，避免工具误判自己在原聊天会话里
- cron agent 创建时设置 `skip_memory=True`，避免 cron 的 system prompt 污染用户画像

不隔离的地方：

- cron 默认跑在 gateway 进程内的线程池里
- 共享同一个 Hermes profile、配置、`.env`、工具、文件系统权限和凭证
- 如果任务配置了 `workdir`，会临时设置进程级 `TERMINAL_CWD`，因此这类任务会被放到单线程顺序执行，避免互相污染

## 日期注入

cron 执行时会给模型的 system prompt 加入日期行，例如：

```text
Conversation started: Tuesday, June 23, 2026
```

该日期来自 `agent/system_prompt.py`，只包含日期，不包含分钟秒。这样做是为了保持同一天内 system prompt 字节稳定，避免破坏 prompt cache。

此外，cron 输出文件中会写入精确运行时间：

```markdown
**Run Time:** 2026-06-23 19:xx:xx
```

这个精确时间用于输出记录，不是提前注入给模型的任务指令。

## 执行结果是否回填主 Agent 上下文

不会自动回填到主 agent 的上下文。

cron 执行完成后的去向是：

```text
cron final_response
  -> 保存 cron output
  -> 保存 cron 独立 session(source='cron')
  -> 可选投递到聊天窗口
  -> 主 agent 上下文不变
```

结果会保存到：

```text
~/.hermes/cron/output/{job_id}/{timestamp}.md
```

同时会写入一个独立 cron session，`source='cron'`，不会 append 到主会话的 messages。

## 用户想基于定时任务结果继续处理怎么办

当前可行方式：

1. 用户回复时带上 cron 输出内容，主 agent 就能基于这段内容继续处理。
2. 用户提供 `job_id`，agent 可以读取 `~/.hermes/cron/output/{job_id}/` 下的最近输出。
3. 如果是 cron 任务之间串联，可以使用 `context_from`，把上游 cron job 的最近输出注入到下游 cron prompt。

更合理的产品设计是增加 “cron delivery follow-up context” 机制：

```text
cron 执行完成
  -> 投递消息时记录 message_id -> job_id / run_session_id / output_path
用户回复这条 cron 消息
  -> gateway 识别 reply_to message_id
  -> 读取对应 cron 输出摘要或全文
  -> 作为当前 turn 的用户上下文注入
  -> 主 agent 基于结果继续执行
```

推荐不要把 cron 输出直接写回主会话历史。更好的方式是只在用户回复 cron 结果的这一轮，把 cron 结果作为当前 user message 的上下文前缀注入。这样既不会污染主 agent 的长期上下文，也不会破坏 system prompt 缓存。
