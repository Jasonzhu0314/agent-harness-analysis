# RAGFlow 表格处理代码梳理

本文基于当前仓库代码梳理 RAGFlow 在表格解析、合并、切分和表头处理上的实现。重点覆盖四类场景：

- 单独的列表头
- 单独的行表头
- 同时有行表头和列表头
- 跨页表格与跨页表头

## 1. 相关入口

RAGFlow 中表格处理主要分两条链路。

### 1.1 结构化表格文件：Excel / CSV / TXT

入口：

- `rag/app/table.py`
- `deepdoc/parser/excel_parser.py`

特点：

- `table parser` 专门处理 Excel / CSV / TXT。
- Excel 会先读取 workbook、sheet、有效行，再解析表头。
- 每一行数据会生成一个 chunk。
- 列可以配置为索引字段、元数据字段或两者都用。

核心代码：

- `rag/app/table.py::Excel.__call__`
- `rag/app/table.py::Excel._parse_headers`
- `rag/app/table.py::chunk`
- `deepdoc/parser/excel_parser.py::RAGFlowExcelParser.html`

### 1.2 文档内表格：PDF / DOCX / Markdown 等

入口：

- `deepdoc/parser/pdf_parser.py`
- `deepdoc/vision/table_structure_recognizer.py`
- `rag/nlp/__init__.py::tokenize_table`

PDF 表格处理流程大致为：

```text
PDF 页面转图片
  -> Layout 模型识别 table 区域
  -> 裁剪 table 图片
  -> TSR 模型识别表格结构
  -> OCR 文本框匹配到行 / 列 / 表头 / 跨行跨列结构
  -> construct_table 构造成 HTML 或描述行列表
  -> tokenize_table 生成 table chunk
```

TSR 指 `TableStructureRecognizer`，识别标签包括：

```python
[
    "table",
    "table column",
    "table row",
    "table column header",
    "table projected row header",
    "table spanning cell",
]
```

其中：

- OCR 负责识别文字内容。
- TSR 负责识别行、列、表头、跨行跨列等结构。

## 2. 切分和入库方式

### 2.1 Excel / CSV 的 table parser

`rag/app/table.py::chunk` 中，每一行数据生成一个 chunk。

例子：

| 姓名 | 年龄 | 部门 |
| --- | --- | --- |
| 张三 | 28 | 销售 |
| 李四 | 31 | 运营 |

会形成类似文本：

```text
- 姓名: 张三
- 年龄: 28
- 部门: 销售
```

以及：

```text
- 姓名: 李四
- 年龄: 31
- 部门: 运营
```

### 2.2 PDF 等文档内表格

`rag/nlp/__init__.py::tokenize_table` 处理表格产物：

- 如果 `rows` 是 HTML 字符串，整个 HTML table 作为一个 table chunk。
- 如果 `rows` 是描述行列表，则默认每 10 行合并成一个 table chunk。
- 如果有表格截图，会把图片也挂到 chunk 上。

描述行列表大致长这样：

```text
姓名：张三; 年龄：28; 部门：销售
姓名：李四; 年龄：31; 部门：运营
```

HTML table 大致长这样：

```html
<table>
  <tr><th>姓名</th><th>年龄</th><th>部门</th></tr>
  <tr><td>张三</td><td>28</td><td>销售</td></tr>
  <tr><td>李四</td><td>31</td><td>运营</td></tr>
</table>
```

## 3. 单独的列表头

### 3.1 表格样式

典型表格：

| 姓名 | 年龄 | 部门 |
| --- | --- | --- |
| 张三 | 28 | 销售 |
| 李四 | 31 | 运营 |

### 3.2 Excel / CSV 处理

Excel / CSV 中，这类表格是支持最稳定的场景。

处理方式：

- 默认第一行是表头。
- 如果有多级表头，会检测前几行是否像表头。
- 多级表头会拼成 `上级-下级` 形式。
- 后续每一行数据生成一个 chunk。

简单表头例子：

```text
- 姓名: 张三
- 年龄: 28
- 部门: 销售
```

多级列表头例子：

| 基本信息 | 基本信息 | 业绩 |
| --- | --- | --- |
| 姓名 | 部门 | Q1 |
| 张三 | 销售 | 100 |

可能会被处理成：

```text
- 基本信息-姓名: 张三
- 基本信息-部门: 销售
- 业绩-Q1: 100
```

相关代码：

- `rag/app/table.py::Excel._parse_simple_headers`
- `rag/app/table.py::Excel._parse_multi_level_headers`
- `rag/app/table.py::Excel._build_hierarchical_headers`

### 3.3 PDF 处理

PDF 中，TSR 会识别 `table column header`。后处理会判断哪些行是 header row：

- 如果单元格有 header 标记，则该行更容易被判为表头。
- 如果大多数数据是数字，而某一行不是数字，也可能被判为表头。

HTML 输出中，表头行会使用 `<th>`：

```html
<table>
  <tr><th>姓名</th><th>年龄</th><th>部门</th></tr>
  <tr><td>张三</td><td>28</td><td>销售</td></tr>
</table>
```

描述行列表输出中，会使用列表头拼接每行：

```text
姓名：张三; 年龄：28; 部门：销售
```

相关代码：

- `deepdoc/vision/table_structure_recognizer.py::construct_table`
- `deepdoc/vision/table_structure_recognizer.py::__html_table`
- `deepdoc/vision/table_structure_recognizer.py::__desc_table`

## 4. 单独的行表头

### 4.1 表格样式

典型表格：

| 项目 | 张三 |
| --- | --- |
| 年龄 | 28 |
| 部门 | 销售 |
| 城市 | 杭州 |

或者没有明确列表头：

| 年龄 | 28 |
| --- | --- |
| 部门 | 销售 |
| 城市 | 杭州 |

### 4.2 当前处理方式

这类表格当前没有专门的“行表头语义展开”逻辑。

在 PDF 的描述行列表模式中，如果列数小于等于 2，代码会倾向于把每行拼成键值式文本：

```text
年龄：28
部门：销售
城市：杭州
```

如果有明确第一行列表头：

| 项目 | 值 |
| --- | --- |
| 年龄 | 28 |
| 部门 | 销售 |

描述行列表可能是：

```text
项目：年龄; 值：28
项目：部门; 值：销售
```

### 4.3 边界

TSR 模型标签里有 `table projected row header`，说明模型层面可以识别候选行表头。但当前 `construct_table` 的后处理主要做两件事：

- 还原 HTML 行列结构。
- 用列 header 生成描述行列表。

它没有稳定地把“第一列是行表头”自动转换成如下语义：

```text
年龄：28
部门：销售
城市：杭州
```

尤其在列数超过 2、或者行表头不是明显键值表时，当前逻辑更偏向保留原表格结构，而不是做语义转置。

## 5. 同时有行表头和列表头

### 5.1 表格样式

典型矩阵表或交叉表：

| 区域 | Q1 | Q2 | Q3 |
| --- | --- | --- | --- |
| 华东 | 100 | 120 | 130 |
| 华南 | 80 | 90 | 110 |

这里：

- 第一行是列表头：`Q1`、`Q2`、`Q3`
- 第一列是行表头：`华东`、`华南`
- 数据格是交叉值：`华东 + Q1 = 100`

### 5.2 当前 HTML 输出

当前更可能保留为 HTML 表格：

```html
<table>
  <tr><th>区域</th><th>Q1</th><th>Q2</th><th>Q3</th></tr>
  <tr><td>华东</td><td>100</td><td>120</td><td>130</td></tr>
  <tr><td>华南</td><td>80</td><td>90</td><td>110</td></tr>
</table>
```

这种输出保留了视觉结构，但没有把每个数据格展开成“行维度 + 列维度 + 值”的长表。

### 5.3 当前描述行列表输出

如果走描述行列表，且 `区域` 被识别成普通列头，可能输出：

```text
区域：华东; Q1：100; Q2：120; Q3：130
区域：华南; Q1：80; Q2：90; Q3：110
```

如果左上角为空，或者第一列行表头没有明确列名，则可能更像：

```text
Q1：100; Q2：120; Q3：130
Q1：80; Q2：90; Q3：110
```

这时 `华东`、`华南` 可能只作为普通单元格被保留，或者因为表头推断不稳定而没有被明确用作每个值的行维度。

### 5.4 没有做的语义展开

当前代码没有看到把矩阵表展开成如下长表结构的逻辑：

```text
区域：华东; 季度：Q1; 值：100
区域：华东; 季度：Q2; 值：120
区域：华东; 季度：Q3; 值：130
区域：华南; 季度：Q1; 值：80
区域：华南; 季度：Q2; 值：90
区域：华南; 季度：Q3; 值：110
```

如果检索重点依赖这种交叉语义，当前实现可能不够强，需要额外加一层矩阵表规范化逻辑。

## 6. 跨页表格和跨页表头

### 6.1 跨页表格样式

第 1 页：

| 姓名 | 年龄 | 部门 |
| --- | --- | --- |
| 张三 | 28 | 销售 |
| 李四 | 31 | 运营 |

第 2 页：

| 姓名 | 年龄 | 部门 |
| --- | --- | --- |
| 王五 | 26 | 技术 |
| 赵六 | 35 | 财务 |

或者第 2 页没有重复表头：

| 王五 | 26 | 技术 |
| --- | --- | --- |
| 赵六 | 35 | 财务 |

### 6.2 PDF 跨页合并

PDF 解析中有“merge table on different pages”逻辑。

合并条件大致为：

- 两个表格不在同一页。
- 页码相邻，页差不超过 1。
- 两段表格之间的垂直距离不太远。
- 前一段没有被标题、caption、reference 等阻断。

合并后，如果一个表格分组跨多个 page，裁图时会逐页裁剪，再纵向拼成一张图。

相关代码：

- `deepdoc/parser/pdf_parser.py::_extract_table_figure`
- `deepdoc/parser/pdf_parser.py` 中 `merge table on different pages` 注释附近
- `deepdoc/parser/pdf_parser.py::cropout`

### 6.3 跨页表头处理

当前代码更像是“跨页表格合并 + 表结构重建”，不是专门的“跨页表头继承”算法。

可能表现：

1. 如果第二页重复了表头：

```text
第 1 页表头：姓名 / 年龄 / 部门
第 2 页表头：姓名 / 年龄 / 部门
```

合并后，TSR/后处理可能把两个表头都识别为 header row。HTML 中可能保留重复表头，或者在描述模式中作为多个 header 行参与后续行描述。

2. 如果第二页没有表头：

```text
第 1 页：姓名 / 年龄 / 部门
第 2 页：王五 / 26 / 技术
```

如果两个分页表被合并到同一个表格分组里，描述模式会优先使用当前行上方最近的 header row，因此有机会让第二页数据继续使用第一页表头。

但这依赖跨页合并成功，以及表头行被正确识别。

3. 如果跨页合并失败：

第二页会作为独立表格处理。如果第二页没有表头，描述行列表可能无法正确补上第一页的列名，只能保留较弱的结构化文本或 HTML 行列。

### 6.4 边界

跨页表格处理不是强语义级别的“续表识别”。它主要依赖版面距离、页码相邻、caption/title 阻断等启发式规则。

因此以下情况容易失败：

- 中间有标题、脚注、页眉页脚干扰。
- 第二页表格位置变化大。
- 续表和前表之间距离超过阈值。
- 第二页没有表头，且跨页合并失败。
- 表头重复但 OCR/TSR 未正确识别。

## 7. 横向表格和旋转表格

PDF 中有自动旋转表格逻辑。

流程：

```text
裁剪表格图
  -> 尝试 0 / 90 / 180 / 270 度
  -> 分别 OCR
  -> 根据 OCR 置信度和文本区域数量选最佳方向
  -> 如果非 0 度明显更好，则使用旋转后的图重新 OCR
```

开关：

```text
TABLE_AUTO_ROTATE=true
```

默认开启。

这可以改善横版扫描表、旋转表格的 OCR 效果。但它解决的是图片方向问题，不解决矩阵表语义展开问题。

## 8. 总结

| 场景 | 当前处理能力 | 主要输出 |
| --- | --- | --- |
| 单独列表头 | 支持较好 | Excel 每行 chunk；PDF HTML 或描述行 |
| 多级列表头 | Excel 支持合并表头拼接；PDF 依赖 TSR | `上级-下级: 值` 或 HTML `<th>` |
| 单独行表头 | 有部分键值表支持，但不稳定 | HTML 保留结构；两列表可能变成键值文本 |
| 行表头 + 列表头 | 保留结构为主，不做长表展开 | HTML table 或 `列头：值` 行描述 |
| 跨页表格 | 有启发式合并 | 合并后裁图拼接、再构造表格 |
| 跨页表头 | 依赖跨页合并和表头识别 | 无专门强继承逻辑 |
| 旋转表格 | 有自动旋转 OCR | 改善 OCR 和 TSR 输入质量 |

如果后续要增强矩阵表检索，建议新增一层表格语义规范化：

```text
识别顶部列表头
  + 识别左侧行表头
  + 遍历每个数据单元格
  -> 展开为 row_header / col_header / value 三元组
```

例如：

```text
区域：华东; 季度：Q1; 值：100
区域：华东; 季度：Q2; 值：120
区域：华南; 季度：Q1; 值：80
```

这类输出比纯 HTML 更适合向量检索和精确问答。
